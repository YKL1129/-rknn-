# RK3588 手语识别优化说明

## 1. 项目现在在做什么

这个项目的主目标是：

- 用摄像头抓取手部关键点。
- 把 30 帧连续动作组成一个时间序列。
- 用分类模型识别当前手语词。
- 把识别出的词拼成一句自然语言。

当前版本里，识别模型输入仍然是 `204` 维：

- `102` 维手部结构特征
- `102` 维速度特征

肌电信号目前没有送入分类模型，只保留采集和显示。

## 2. 这次修改了什么

### 2.1 训练部分

原来训练端更像一个简单 MLP。
现在改成了轻量时序模型：

- `Conv1D + BatchNormalization + GlobalAveragePooling1D + Dense`

这样做的目的：

- 保留 `(30, 204)` 输入格式，不破坏 RK3588 部署链路
- 比单纯拍平后全连接更适合连续动作
- 参数量仍然比较轻，适合后续导出给 RK3588

同时增加了：

- 训练集 / 验证集 / 测试集分层切分
- `EarlyStopping`
- `ModelCheckpoint`
- ONNX 导出
- 离线评估脚本 `eval_model.py`

### 2.2 识别部分

识别端不再只看“最大概率是不是超过阈值”，而是增加了双重判断：

- 第一名概率超过阈值
- 第一名和第二名的概率差 `margin` 也要超过阈值

这样可以减少相似手势之间的误判。

同时把词汇确认逻辑改成了状态机：

- `IDLE`
- `CANDIDATE`
- `CONFIRMED`
- `COOLDOWN`

目的：

- 减少同一个词被重复入库
- 减少抖动触发
- 更适合连续识别

### 2.3 连句部分

增加了本地模板和缓存：

- `phrase_templates.json`
- `phrase_cache.json`

优先级现在是：

1. 先查模板
2. 再查缓存
3. 最后才调用大模型

这样做的好处：

- 固定展示语料几乎零延迟
- 减少本地小模型胡乱补词
- 大模型只负责兜底润色

### 2.4 骨架显示

现在默认会显示“简化骨架”：

- 比完整 MediaPipe 骨架更干净
- 更容易直接看出有没有抓到关键点
- 对性能压力也更小

如果想切回完整骨架，可以改 `runtime_config.json`：

```json
"draw_landmarks": true,
"draw_full_landmarks": true
```

如果不想显示骨架：

```json
"draw_landmarks": false
```

### 2.5 训练卡住问题

这次额外补了训练 UI 状态：

- 训练开始时禁用按钮
- 显示 epoch 进度
- 训练异常会弹错误框
- 训练完成后恢复按钮

所以现在如果训练慢，你能看到它是在跑；
如果训练报错，也不会表现成“像卡死了一样没反应”。

## 3. 新增文件说明

- `gesture_runtime.py`
  运行时公共工具，包含配置、状态机、模板翻译、数据切分。

- `runtime_config.json`
  运行时配置文件，控制阈值、margin、简骨架、分辨率等。

- `phrase_templates.json`
  固定展示语料模板。

- `eval_model.py`
  离线评估脚本，输出整体准确率、每类准确率、混淆矩阵。

## 3.1 训练集和测试集怎么录

现在支持固定数据分区录制：

- `MP_Data_Fusion/train/词/序号`
- `MP_Data_Fusion/test/词/序号`

录制时界面上会显示一个 `DATA SPLIT` 按钮：

- `TRAIN` 表示当前录到训练集
- `TEST` 表示当前录到固定测试集

这样做以后：

- 训练默认优先读取 `train`
- `eval_model.py --split test` 默认优先读取固定测试集
- `eval_model.py --split holdout` 才是老的随机切分评估

推荐录法：

1. 大部分样本录到 `train`
2. 每个词单独留一批从未参与训练的样本录到 `test`
3. `test` 录制时尽量换一点距离、速度、角度或光照

## 3.2 新的母版构建流程

现在推荐的新流程是：

1. 在界面里录制视频母版
2. 母版会保存到：
   - `Gesture_Master/train/...`
   - `Gesture_Master/test/...`
3. 录完之后手动运行构建脚本，把母版转成当前模型需要的特征集
4. 训练和评估优先使用构建后的输出目录

示例命令：

```powershell
python build_dataset.py --母版目录 Gesture_Master --输出目录 MP_Data_Build --数据分区 all --重建全部
```

如果只想处理新增样本：

```powershell
python build_dataset.py --母版目录 Gesture_Master --输出目录 MP_Data_Build --仅增量
```

说明：

- `build_dataset.py` 参数、日志、错误信息默认都是中文
- 旧版 `204` 维 `.npy` 会以 `legacy_feature` 形式登记到清单里，作为过渡资产保留
- 真正长期可复用的源数据是视频母版，而不是最终特征帧

## 4. 你看到的验证结果该怎么理解

你运行得到的结果是：

- 测试样本数 `32`
- 整体准确率 `1.0000`
- 混淆矩阵全对角

这说明：

- 模型在当前离线测试集上已经能把这 10 类区分开
- ONNX 导出后的分类结果没有明显错误

但它**不等于实机一定同样 100%**，原因有三个：

1. 测试集太小  
   例如 `老师` 只有 `1` 条测试样本，统计不稳定。

2. 数据分布很近  
   训练集和测试集都来自同一批采集，人物、机位、光照、背景差异不够大。

3. 离线分类不等于连续识别  
   真实场景还有误触发、重复触发、动作切换、停顿判断这些问题。

所以这个结果更准确的解释是：

- “模型本身已经学会了当前数据集”
- 但“连续实机效果还需要继续验证”

## 5. 下一步怎么验证最有价值

建议按这个顺序测：

1. 离线继续补独立测试集  
   最好换时间、换距离、换背景、换速度。

2. RK3588 实机连续测试  
   每个词连续打 20 次，统计：
   - 正确次数
   - 误触发次数
   - 重复入库次数
   - 平均延迟

3. ONNX 和 RKNN 一致性测试  
   确认 PC 端和 RK3588 端对同一批输入输出一致。

## 6. 常改的参数在哪

都在 `runtime_config.json`：

- `confidence_threshold`
  分类置信度阈值

- `margin_threshold`
  第一名和第二名概率差阈值

- `confirm_frames`
  连续多少次确认才算真正入库

- `cooldown_seconds`
  同一词确认后冷却多久

- `sentence_idle_seconds`
  多久没有新词就触发连句

- `draw_landmarks`
  是否显示骨架

- `draw_full_landmarks`
  `false` 是简骨架，`true` 是完整骨架

- `process_width` / `process_height`
  MediaPipe 实际处理分辨率

## 7. 目前仍然存在的限制

- 肌电没有重新纳入模型
- 模板语料还比较少
- RK3588 端还没有做专门的 ONNX/RKNN 一致性统计工具
- 离线 100% 结果目前还不能代表跨人泛化能力

## 8. 一句话总结

这次修改的重点不是“把数字调高一点”，而是把整个系统从“能跑”往“能解释、能验证、能部署、能演示”推进了一步。
